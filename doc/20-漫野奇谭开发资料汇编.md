## 1. 整体架构

### 1.1 技术栈

| 层面 | 技术 |
|------|------|
| 引擎 | **LibGDX**（Java 开源框架），非 Unity |
| 语言 | Java（逻辑层） |
| 数据格式 | **XML**（所有叙事内容、角色特质、变形、事件） |
| 美术资源 | PNG（分层精灵图） |
| 模组分发 | Steam Workshop |

### 1.2 数据驱动架构

漫野奇谭的核心设计哲学是**数据与逻辑分离**：

- **数据层（XML）**：所有故事事件、角色 Aspect（特质/增益/减益）、变形规则、遭遇定义均以 XML 文件存储
- **逻辑层（Java）**：引擎代码只负责**解释** XML 定义，不硬编码任何叙事内容
- 内容作者（含模组开发者）可以创建完整的叙事线而**无需编写任何 Java 代码**

### 1.3 模组系统

```
myMod/
  content/
    events/          # 事件 XML
      myEvent1.xml
    aspects/         # Aspect 定义
      myAspects.xml
    transformations/ # 变形规则
      myTransformation.xml
  assets/
    images/          # 背景图、精灵层
  text/
    en/              # 本地化
      strings.xml
  mod.json           # 元数据：名称、版本、作者、依赖
```

- 游戏启动时扫描模组目录，合并到主数据注册表
- 使用命名空间约定避免 ID 冲突（如 `myMod_customAspect`）
- 内置编辑器：可通过开发者工具打开 GUI 编辑器，实时预览并一键上传 Steam Workshop

## 2. 演出模式（核心）

### 2.1 漫画分镜式呈现

漫野奇谭的演出模式本质是一个**运行时组装的动态漫画系统**：

```
┌─────────────────────────────────────────────┐
│  事件被触发                                   │
│     ↓                                        │
│  按事件定义组装场景序列（Scene/Panel）          │
│     ↓                                        │
│  Panel 1 ──→ Panel 2 ──→ ... ──→ Panel N    │
│  [背景]       [选择A]                        │
│  [角色站位]   [选择B]                        │
│  [对话/旁白]                                 │
│  [视觉效果]                                  │
│     ↓                                        │
│  最终后果应用到游戏状态                        │
└─────────────────────────────────────────────┘
```

### 2.2 Panel（面板）结构

每个 Panel 包含：

| 元素 | 说明 |
|------|------|
| **Background** | 环境背景图（森林、城镇、战场等） |
| **Actor Placement** | 角色精灵位置（左/中/右、前景/背景） |
| **Dialogue** | 角色对话，支持条件文本变体 |
| **Narration** | 全知旁白文本 |
| **Choice** | 玩家选择点，分支到不同后续 Panel |
| **Effect** | 触发后果（赋予 Aspect、修改关系、触发变形） |

**关键特性**：Panel 不是预渲染的静态画面，而是**运行时动态组装**的 —— 同一事件在不同游玩中看起来完全不同，因为：
- 参与角色不同（通过 Role 系统动态分配）
- 角色外观不同（反映当前变形状态：伤疤、断臂、魔法改变、衰老）
- 关系与情绪状态不同

### 2.3 演出流程

```
1. 事件被选中并开始执行
2. 第一个 Scene/Panel 被组装并展示
3. 玩家阅读对话和旁白，在提示时做出选择
4. 基于选择，下一个 Panel 被组装
5. 循环直到所有 Panel 展示完毕
6. 最终后果应用到游戏状态
```

### 2.4 角色站位系统

角色在 Panel 中的位置由事件 XML 指定，典型布局：

```
┌───────────────────────────────────┐
│  背景：幽暗森林                    │
│                                   │
│   [角色A]    [角色B]    [角色C]    │
│    左侧       中央       右侧      │
│                                   │
│   ─── 对话气泡 ───                 │
│   角色B："有东西在看我们..."        │
│                                   │
│   [▶ 拔出武器]   [▶ 转身逃跑]      │
└───────────────────────────────────┘
```

## 3. 事件/故事系统

### 3.1 事件管线架构

```
注册 → 条件评估 → 加权选择 → 执行 → 后果应用
```

**详细流程：**

1. **注册**：游戏 + 模组的所有事件加载到事件注册表，每个有唯一 ID
2. **条件评估**：在每个 Hook 触发点，过滤所有事件：
   - Aspect 要求：角色是否拥有/不拥有特定 Aspect？
   - 属性要求：角色属性是否在指定范围？
   - Role 要求：能否找到合适的角色填充事件的 Role？
   - Tag 要求：战役/角色是否拥有必要的叙事标签？
   - 概率：对事件的基础概率进行随机检定
3. **加权选择**：合格事件按评分排名：
   - 基础权重
   - **新鲜度惩罚**：近期触发过的事件被降权
   - **关联度加成**：与当前角色弧线相关的事件得分更高
   - **战役节奏**：系统在战役时间线上平衡事件类型
   - **角色历史**：建立在已有关系上的事件得分更高
4. **执行**：胜出事件被实例化，Role 被最合适的角色填充
5. **后果应用**：所有指定效果被执行

### 3.2 事件 XML 结构

```xml
<event id="eventName">
  <!-- 前置条件 -->
  <criteria>
    <requiredAspect>warrior</requiredAspect>
    <requiredTag>chapter2</requiredTag>
    <chance>0.3</chance>
  </criteria>

  <!-- Role 定义 — 角色槽位 -->
  <roles>
    <role id="hunter" min="1" max="1" />
    <role id="warrior" min="1" max="1" />
    <role id="friend" min="0" max="1" />
  </roles>

  <!-- 场景序列 -->
  <scene id="intro">
    <background>forest_dark</background>
    <narration>树木在你周围合拢...</narration>
    <dialogue actor="hunter">有东西在看我们。</dialogue>
    <choice target="fight">拔出武器</choice>
    <choice target="flee">转身逃跑</choice>
  </scene>

  <scene id="fight">
    <narration>刀光闪烁...</narration>
    <effect>
      <grantAspect target="warrior">scarred</grantAspect>
      <modifyRelationship actor1="hunter" actor2="warrior" delta="1"/>
    </effect>
  </scene>

  <scene id="flee">
    <narration>你们狂奔而去...</narration>
  </scene>
</event>
```

### 3.3 Hook（触发点）类型

事件注册到特定 Hook 触发：

| Hook | 触发时机 |
|------|----------|
| `onCampaignStart` | 新战役开始 |
| `onEncounterEnd` | 战术战斗结束 |
| `onMonthEnd` | 每月末（大地图） |
| `onChapterEnd` | 章节过渡 |
| `onMissionStart` | 战术任务开始前 |
| `onDeath` | 角色死亡 |
| `onHit` / `onTurnStart` | 战斗中 |
| `onArriveAtSite` | 到达地点 |

### 3.4 Role（角色槽位）系统

事件定义**抽象 Role**而非具体角色：

```
事件定义: "需要一个猎人、一个战士、一个朋友"
运行时填充: "张三是猎人 → 填入hunter角色，李四是战士 → 填入warrior角色"
```

Role 填充依据：
- **职业要求**（warrior Role 需要战士职业角色）
- **关系要求**（lover Role 需要与主角有恋情关系的角色）
- **Aspect 要求**（scarredVeteran Role 需要 scarred Aspect）
- **Tag 要求**（previousEncounter Role 需要特定故事标签）

## 4. 角色变形系统

### 4.1 分层精灵渲染

角色渲染采用**精灵图层堆叠**：

```
┌─ 角色精灵栈 ─────────────┐
│  装备层 (Equipment)       │
│  覆盖层 (Overlay)         │ ← 魔法纹路、伤疤等
│  头部层 (Head)            │
│  躯干层 (Torso)           │
│  手臂层 (Arms)            │ ← 断臂时替换为空/魔法臂
│  腿部层 (Legs)            │
│  基础身体 (Base Body)     │
└───────────────────────────┘
```

每一层可以独立替换或修改。变形通过**替换或添加精灵层**实现。

### 4.2 变形类型

| 类型 | 例子 | 机制 |
|------|------|------|
| **物理伤害** | 断臂、失去眼睛、伤疤 | 替换对应肢体/头部精灵层 |
| **魔法变形** | 魔法臂（替换断肢为发光灵体臂）、狼头、树化、结晶化 | 替换层 + 添加特效覆盖 |
| **衰老** | 头发灰白、姿态改变 | 随战役时间线渐进修改层 |
| **装饰添加** | 纹身、战妆、魔法印记 | 添加覆盖层 |

### 4.3 变形应用流程

```
1. 事件后果指定变形：<applyTransformation target="warrior" transformation="lostArm"/>
2. 变形定义指定修改哪些精灵层
3. 角色视觉表示立即更新
4. 变形同时赋予/修改 Aspect（断臂 → oneArmed Aspect → 影响战斗属性）
5. 变形永久持续，在后续所有漫画面板和战术战斗中可见
```

### 4.4 变形定义格式（概念性 XML）

```xml
<transformation id="lostArm">
  <layer target="leftArm" replaceWith="empty"/>
  <grantAspect>oneArmed</grantAspect>
  <statModifier>meleeAccuracy -2</statModifier>
</transformation>
```

### 4.5 叙事整合

变形**永远不是随机装饰**——总是与叙事事件绑定：
- 断臂来自特定的故事时刻，游戏会记住
- 角色传记记录事件
- 未来事件可引用变形（需要 oneArmed Aspect 的事件），创造叙事回呼
- 这是漫野奇谭"角色承载历史"设计的核心

### 4.6 Legacy（传承）系统

- 存活过一场战役的英雄可成为**传承英雄**
- 变形、Aspect、关系、故事标签**跨战役保留**
- 一场战役中失去手臂的角色，在后续战役中被招募时仍然只有一只手臂

## 5. 叙事标签系统

### 5.1 标签定义

**标签（Tags）** 是附加在实体上的字符串标签，用于追踪叙事状态：

| 标签类型 | 附加对象 | 例子 |
|----------|----------|------|
| 角色标签 | 单个英雄 | `survivedFire`、`metTheHermit`、`lostArmInBattle3` |
| 战役标签 | 全局战役 | `chapter2Complete`、`darknessRising` |
| 地点标签 | 地图位置 | `liberated`、`corrupted`、`siteOfBattle3` |

### 5.2 标签的三大用途

**1) 事件前置条件**
```xml
<requiredTag>metTheHermit</requiredTag>
```
只有拥有此标签的战役/角色才能触发该事件。

**2) 事件后果赋予**
```xml
<grantTag target="hunter">survivedFire</grantTag>
```
事件完成后赋予标签，创造叙事连续性。

**3) 条件文本**
```xml
<narration>
  <text if:hasTag="survivedFire">又一次，火焰。你想起索恩谷那场大火。</text>
  <text>火焰在黑暗中跳动。</text>
</narration>
```
根据标签状态显示不同文本。

### 5.3 Tag 与 Aspect 的区别

| | Tag（标签） | Aspect（特质） |
|--|------------|----------------|
| **性质** | 纯叙事，无游戏机制效果 | 有机制效果（属性修正、Hook 处理、战斗效果） |
| **用途** | 故事追踪、叙事门控 | 游戏性追踪 |
| **关系** | 通常与 Aspect 配对使用 | 可独立存在 |

实践中，一次变形通常同时赋予两者：
- 机制 Aspect（如 `oneArmed`，影响战斗属性）
- 叙事 Tag（如 `lostArmInBattle3`，供故事引用）

## 6. 模组与配置格式

### 6.1 文件格式概览

| 用途 | 格式 | 说明 |
|------|------|------|
| 事件定义 | XML | 完整的场景序列、条件、后果 |
| Aspect 定义 | XML | 特质/增益/减益、Hook、属性修正 |
| 变形定义 | XML | 精灵层替换规则 |
| 精灵资源 | PNG | 分层结构，按目录组织 |
| 本地化 | XML/键值对 | 多语言支持 |
| 模组元数据 | JSON | 名称、版本、作者、依赖 |

### 6.2 Aspect 完整定义示例

```xml
<aspect id="myMod_customAspect">
  <tag>myMod</tag>
  <description>来自模组的自定义特质</description>
  <hook id="onTurnStart">
    <effect>
      <modifyStat target="self" stat="speed" delta="1"/>
    </effect>
  </hook>
  <conflictsWith>cursed</conflictsWith>
  <stackLimit>3</stackLimit>
</aspect>
```

**Aspect 能力：**
- 关联 Hook（在特定时机触发效果）
- 修改属性值
- 与其他 Aspect 冲突
- 支持堆叠（如 `battleHardened` 最多叠 3 层，每层 +1 准确度）
- 可查询/可否定（`<requiredAspect>!immortal</requiredAspect>`）

### 6.3 事件权重与节奏控制

```xml
<event id="myEvent" weight="5">
  <criteria>
    <cooldown>30</cooldown>        <!-- 冷却：30天内不再触发 -->
    <maxPerCampaign>3</maxPerCampaign>  <!-- 每战役最多3次 -->
    <chapter min="2" max="4"/>     <!-- 仅在2-4章触发 -->
  </criteria>
</event>
```

节奏控制机制：
- **Cooldown**：触发后的冷却期
- **MaxPerCampaign**：单战役触发上限
- **Chapter Range**：章节范围限制
- **新鲜度惩罚**：近期触发过的事件自动降权
- **类型平衡**：系统自动平衡战斗/社交/探索/个人事件的比例

## 7. 设计原则总结

### 7.1 六大核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **数据优于代码** | 几乎所有叙事内容以 XML 存储，非程序员也能创作 |
| 2 | **角色槽优于具体角色** | 事件定义抽象 Role，运行时动态填充 |
| 3 | **双轨状态追踪** | Aspect（机制）+ Tag（叙事）并行追踪 |
| 4 | **加权选择 + 新鲜度偏好** | 事件选择保证多样性，尊重叙事关联和战役节奏 |
| 5 | **视觉 = 叙事** | 角色变形总是故事驱动的，视觉上永久持续 |
| 6 | **传承持久化** | Legacy 系统让变形、关系、故事事件跨战役延续 |

### 7.2 演出模式的设计要点

1. **运行时组装**：Panel 不是预制的，而是根据当前角色状态动态生成
2. **角色即历史**：角色外观是其经历的可视化记录
3. **选择即叙事**：玩家选择直接影响后续 Panel 组装和后果
4. **回呼系统**：Tag 机制让后续事件能引用和构建在先前事件之上
5. **多线叙事**：Role 系统允许任意合格角色参与，创造高重玩性

## 8. 与本项目（市井录）的对照

### 8.1 已实现的对应系统

| 漫野奇谭系统 | 市井录对应 | 文件 |
|-------------|-----------|------|
| 事件 XML 定义 | TypeScript 场景数据（L0/L1/L2） | `src/data/sceneLibrary/l0/*.ts` |
| Role 系统 | 目标条件（npcConditions） | `src/ai/sceneLibrary/types.ts` |
| Aspect 系统 | NarrativeTag + CharacterTransformation | `src/ai/sceneLibrary/types.ts` |
| Tag 系统 | NarrativeTag（字符串标签） | `src/ai/sceneLibrary/types.ts` |
| 变形系统 | CharacterTransformation | `src/ai/sceneLibrary/types.ts` |
| 条件文本 | `{if hasTag:xxx}...{/if}` 模板 | `src/ai/sceneLibrary/narrativeFormatter.ts` |
| 事件匹配 | Matcher（多层级条件评分） | `src/ai/sceneLibrary/matcher.ts` |
| 结果判定 | Resolver（certain/chance/contested） | `src/ai/sceneLibrary/resolver.ts` |
| 漫画面板呈现 | 活跃场景管理器（多步骤场景） | `src/ai/sceneLibrary/activeSceneManager.ts` |
| 新鲜度/冷却 | Cooldown 管理 | `src/ai/sceneLibrary/sceneLibraryManager.ts` |
| Legacy 系统 | 无直接对应（单人原型） | — |

### 8.2 关键差异

| 方面 | 漫野奇谭 | 市井录 |
|------|----------|--------|
| 规模 | 数十英雄，回合制战役 | 万级 NPC，实时模拟 |
| AI 层级 | 单层（故事事件驱动） | 三层 LOD（GOAP / 行为树 / 统计模拟） |
| 玩家角色 | 控制英雄小队 | 单一角色融入 NPC 世界 |
| 视觉表现 | 2D 精灵分层漫画 | 文字为主 + 水墨 UI |
| 内容格式 | XML（非程序员友好） | TypeScript（开发者友好） |
| 场景数量 | 数百官方 + 无限模组 | ~420 预定义 + 动态生成 |

### 8.3 可借鉴的改进方向

1. **数据驱动化**：考虑将场景数据从 TypeScript 迁移到 JSON/YAML，方便非程序员创作
2. **视觉演出**：漫野奇谭的漫画分镜式呈现是核心体验，市井录可在文字叙事中增加类似分镜的结构感
3. **传承系统**：长线游玩中，NPC 的历史经历应更可视化
4. **模组接口**：预留模组扩展接口，允许外部注入场景/事件
5. **编辑器工具**：开发可视化场景编辑器，降低内容创作门槛
