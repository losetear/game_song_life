# 05 — NPC 与 AI 系统

## 核心理念

世界由上万实体组成，NPC有自己的日子要过。但10000个实体不可能都跑完整AI — 需要**分层架构**，让离玩家近的NPC有深度，远处的NPC用简化模拟。

> **架构细节详见 [11-技术架构.md](11-技术架构.md) 的"分层AI系统"章节。本文档侧重AI的行为设计。**

## 分层AI概览

| 层级 | 技术 | 适用对象 | 数量上限 | 更新频率 |
|------|------|---------|---------|---------|
| L0 | GOAP | 玩家视野内的核心NPC | ≤10 | 每次玩家操作 |
| L1 | 行为树 | 同区域其他NPC + 家畜 | ≤500 | 每时辰 |
| L2 | 统计模拟 | 远处所有实体 | ≤10000 | 每天 |

## L0 — GOAP（核心NPC深度AI）

仅用于玩家视野内最重要的NPC（≤10个），保证涌现质量。

### NPC 状态（World State per NPC）

| 状态维度 | 示例值 | 说明 |
|---------|--------|------|
| 位置 | 夜市、家中、农田… | NPC在哪里 |
| 饥饿度 | 0-100 | 需要吃饭 |
| 疲劳度 | 0-100 | 需要休息 |
| 金钱 | 数值 | 可用于交易 |
| 持有物品 | 列表 | 随身携带物 |
| 心情 | 数值 | 影响行为选择 |
| 好感度表 | {npcId: 数值} | 对其他人的态度 |
| 职业 | 身份 | 影响可用行动 |
| 当前目标 | 目标栈 | 当前最优先要做什么 |

### NPC 目标体系

**生存目标（最高优先级）**
- 饥饿了 → 找食物
- 累了 → 回家休息
- 生病了 → 看大夫
- 被攻击 → 逃跑/反击

**日常目标（按职业和性格分配）**
- 商贩：进货 → 摆摊 → 卖货 → 收摊
- 农夫：耕种 → 浇水 → 收割 → 赶集买卖
- 衙役：巡逻 → 检查 → 抓贼 → 交接班
- 书生：读书 → 作文 → 偶尔社交
- 大夫：坐堂 → 看诊 → 采药
- 小偷：踩点 → 找目标 → 偷窃 → 销赃 → 躲藏
- 厨子：备料 → 做菜 → 卖食 → 收摊

**长期目标（影响人生轨迹）**
- 攒钱买房
- 科举中举
- 找对象结婚
- 报仇雪恨
- 成为行业翘楚

### NPC 性格模型

每个NPC有性格标签组合，影响决策权重：

| 性格标签 | 特点 | 行为倾向 |
|---------|------|---------|
| 勤劳 | 优先工作 | 工作相关行动权重+50% |
| 懒惰 | 拖延摸鱼 | 休息娱乐权重+50% |
| 贪婪 | 追求利益 | 金钱相关行动权重+50% |
| 善良 | 帮助他人 | 利他行为权重+50% |
| 狡猾 | 精于算计 | 欺骗类选项开放 |
| 暴躁 | 容易发怒 | 冲突概率增加 |
| 胆小 | 回避危险 | 逃跑概率增加 |
| 好客 | 乐于交友 | 主动社交概率增加 |

> 一个NPC通常有2-3个性格标签组合，如"勤劳+善良"或"狡猾+贪婪"。

### NPC 记忆系统（原稿缺失，补充）

每个NPC维护一份记忆日志，影响其行为决策：

| 记忆类型 | 保存时间 | 示例 | 影响 |
|---------|---------|------|------|
| 近期事件 | 3-7天 | "三天前张三偷了我的钱" | 直接影响对某人态度 |
| 长期记忆 | 永久 | "去年冬天那场大雪冻死很多人" | 偶尔在对话中提及 |
| 人物印象 | 永久（缓慢衰减） | "那个卖炊饼的伙计人不错" | 影响好感度变化速度 |
| 交易记录 | 30天 | "上周从东市进了一批布，花了两贯" | 影响定价和进货决策 |
| 恩怨记录 | 永久 | "这人救过我的命" / "这人害我坐牢" | 特殊交互选项解锁 |

**记忆衰减规则**：
- 近期事件随时间转为长期记忆（细节模糊）
- 对未再接触的人，好感度每30天自然衰减5点
- 恩怨记录不会衰减，但严重程度可随时间略微降低

### GOAP 行动表（原稿缺失，补充）

每个行动需要定义：前置条件、执行效果、消耗代价

```
行动示例：

【买炊饼】
  前置：位置=街边/夜市 AND 金钱>=10 AND 饥饿度<100
  效果：饥饿度+30, 金钱-10, 持有物品+炊饼x1
  消耗：时间-0.3时辰
  可用者：所有人

【偷窃】
  前置：位置=暗处/拥挤处 AND 性格包含"狡猾" OR 职业=小偷
  效果：成功(60%)→金钱+目标身上钱*0.5, 失败(40%)→被通缉
  消耗：时间-1时辰
  可用者：小偷、狡猾角色

【坐堂看病】
  前置：职业=大夫 AND 位置=药铺 AND 时辰=巳-未
  效果：金钱+30~100, 声望+1
  消耗：时间-2时辰
  可用者：大夫

【巡逻】
  前置：职业=衙役 AND 时辰=辰-戌 AND 位置=城内
  效果：发现犯罪概率+50%, 声望+1
  消耗：时间-2时辰, 疲劳+20
  可用者：衙役
```

> **设计要点**：行动表初期控制在 30-50 个行动以内，后续按职业逐步扩展。每个行动都需要明确的数值定义，避免模糊描述。

## L1 — 行为树（同城NPC简化AI）

用于玩家同区域但不在视野内的NPC（≤500个）。牺牲涌现深度换取性能。

### 行为树模板（按职业）

```
[通用NPC行为树骨架]
  Root → Selector
  ├── 满足生存需求？→ 执行生存行动
  ├── 工作时间？     → 执行职业行动
  ├── 社交时间？     → 执行社交行动（随机概率）
  └── 默认          → 待在原地/回家

[商贩模板]
  工作行动 = Selector
    ├── 库存充足 → 卖货
    ├── 库存不足 → 进货 → 卖货
    └── Random(15%) → 去茶馆

[农夫模板]
  工作行动 = Selector
    ├── 春季 → 播种
    ├── 夏季 → 浇水/除草
    ├── 秋季 → 收割
    └── 冬季 → 休息/赶集

[衙役模板]
  工作行动 = Selector
    ├── 发现犯罪 → 追捕
    └── 正常 → 巡逻（按路线点移动）

[小偷模板]
  工作行动 = Selector
    ├── 有目标且无人看 → 偷窃
    ├── 无目标 → 踩点（在人群中游荡）
    └── Random(20%) → 去销赃
```

### L1 NPC不维护的项目（与L0的区别）

| 移除项 | 原因 | 替代方案 |
|--------|------|---------|
| 完整记忆系统 | 500个NPC各持记忆太贵 | 仅保留3条近期事件 |
| GOAP规划器 | 太慢 | 行为树（毫秒级） |
| 动态对话生成 | 无需对话 | 预设对话模板 |
| NPC间好感度 | 太多关系对 | 仅保留与玩家和L0 NPC的关系 |

### L1 NPC状态更新

```
每时辰（非每动作）批量更新：
  1. 按职业分组
  2. 每组批量计算行为树
  3. 批量更新位置和状态
  4. 如有重要结果（犯罪被抓等），推送至事件队列

预计性能：500个NPC × 行为树 ~ 0.01ms/个 = 5ms/时辰
```

## L2 — 统计模拟（远处实体）

城外和远处的实体不模拟个体行为，只做区域级统计。

### L2 统计规则

```
[区域: 城外农田]
  每天计算一次：
    区域内农夫数 = 从L1/L2人口统计获取
    区域面积 = 固定值
    天气修正 = 从WeatherSystem获取
    季节修正 = 查表

    产量 = 面积 × 基础亩产 × 天气修正 × 季节修正 × 劳力修正
    农夫消耗 = 农夫数 × 日消耗
    净产出 = 产量 - 消耗

    粮食供给 → 推入EconomySystem
    如果 净产出 < 0 → 部分农夫可能迁入城中

[区域: 山林]
  每天计算一次：
    野生动物数 = 上日数量 × (1 + 繁殖率 - 狩猎率 - 自然死亡率)
    如果 野生动物数 > 容量上限 → 设为上限（防止无限增长）
    如果 野生动物数 < 最小值 → 触发"此山野兽稀少"消息

    可采集资源 = 基于季节和天气计算
    采集后按规则重生（不同资源不同重生时间）

[区域: 河流]
  每天计算一次：
    鱼获量 = 基于季节 + 捕鱼人数
    水匪活动 = 基于治安值随机
```

### L2 → L0/L1 的升级触发

当玩家移动到某个区域时，该区域的L2实体需要"具象化"为L0/L1：

```
具象化流程：
  1. 读取区域统计数据（如"山林有15只兔子"）
  2. 按规则在Grid中放置具体实体（15只兔子分布在合适位置）
  3. 为每个实体创建ECS组件（Vital, Growth等）
  4. 核心NPC升级为L0 GOAP
  5. 其他NPC/动物升级为L1 行为树

离开时反序列化：
  1. 统计L0/L1实体的结果（兔子现在剩12只了）
  2. 销毁ECS组件，释放内存
  3. 将统计数据写回L2
```

> **关键**：具象化必须确定性（基于种子），这样玩家反复进出同一区域，实体分布一致。

### NPC 日程（基于GOAP动态生成）

NPC不按固定脚本走，但会有习惯倾向：

```
[商贩王老板 — 勤劳+贪婪]
  卯时 → 起床，去东市进货
  辰时 → 回铺子开张
  巳-午 → 卖货，与客人交谈
  未时 → 午饭，算账
  申-酉 → 继续营业
  戌时 → 关门，去酒楼社交
  亥时 → 回家睡觉

  * 特殊：如果听说哪有好货，可能临时去进货
  * 特殊：如果被人欠钱，可能去找人讨债
  * 特殊：如果听说夜市热闹，可能晚关门去摆摊
```

## 涌现事件示例

NPC的独立决策会导致意想不到的事件涌现：

### 示例1：小偷被抓
```
1. 张三（混混）目标：偷钱
2. 张三在夜市踩点，看到喝醉的客商
3. 张三偷了客商的钱袋
4. 钱捕头刚好巡逻经过
5. 客商发现丢钱大喊
6. 钱捕头追捕张三
7. 张三被抓入狱

→ 玩家可能目击全过程
→ 玩家可能帮助追贼（提升声望）
→ 玩家可能在张三被抓前被张三栽赃
```

### 示例2：粮价飞涨
```
1. 夏季暴雨，秋收减产（系统事件）
2. 城外农户收成不好（NPC农夫状态更新）
3. 粮铺进货价格上涨
4. 王掌柜（贪财）提高粮价
5. 市民抱怨，部分人囤粮
6. 官府出面平抑粮价
7. 有人趁乱偷粮

→ 玩家如果有存粮可以高价卖出
→ 玩家如果种田，今年收入减少
→ 玩家可以报官/帮忙维持秩序
```

### 示例3：瘟疫传播
```
1. 某NPC从外地来，携带疾病
2. 接触过的人都可能感染
3. 大夫发现异常，发出警告
4. 感染者增多，城里人心惶惶
5. 官府封城/隔离
6. 药铺药品紧缺，价格飞涨
7. 大夫研制药方

→ 玩家可能被感染（取决于接触史）
→ 玩家可以帮忙采药/送药
→ 玩家可以趁机囤药高价卖（但声望下降）
```

## NPC对话生成规则

### 对话结构
```
[问候/反应] + [内容] + [暗示/需求]

示例：
"这位客官，今儿个来得早啊。"
（王掌柜看到玩家，根据时间/关系/心情生成问候）

"最近城里不太平，您出门可要当心。"
（基于王掌柜知道的消息，善意提醒）

"对了，我那儿有批上好的苏绣，要不要看看？"
（王掌柜当前目标：卖货，所以主动推销）
```

### 对话影响因素
| 因素 | 影响 |
|------|------|
| 好感度 | 高→热情；低→冷淡/拒绝交谈 |
| 时间 | 早上→问候"早"；深夜→奇怪为什么还在 |
| 天气 | 雨天→抱怨天气；晴天→心情好 |
| 近期事件 | 发生大事→谈论大事 |
| NPC当前目标 | 目标相关→主动提及 |
| 玩家声望 | 高声望→恭敬；低声望→轻视 |

## 涌现叙事系统（演出库架构）

> **核心设计理念参考《漫野奇谭》(Wildermyth)**：目标≠固定行为。目标是一种意图方向，实现路径应该多样化且受情境影响。系统从演出库中匹配演出，而不是机械地映射一个行动。

### 设计哲学

《漫野奇谭》的设计师 Nate Austin 称这套方法为"戏剧的图书馆"(Library of Plays)：

- 故事作者编写一幕幕**演出**，组成演出库
- 每个演出标记需要特定的**角色特征**参与
- 当NPC进入一个情节点时，系统从演出库**搜寻匹配**的演出
- 演出有自己的参数，只要"演员"就位，马上就可以开演

**关键转变**：NPC有了"觅食"目标后，不应该总是执行"买炊饼"。目标→行为之间增加一层"演出匹配"，使得同一个目标在不同性格/情绪/关系/地点下产生截然不同的行为和叙事。

### 三层架构：目标 → 演出 → 行动

| 层级 | 来源 | 示例 |
|------|------|------|
| **目标** (Goal) | 需求+性格+情绪生成 | "觅食""社交""赚钱""休息" |
| **演出** (Scene) | 目标+性格+情境从演出库匹配 | "街边偶遇老友""茶楼打听消息""暗巷偷窃" |
| **行动** (Action) | 演出中实际执行的效果 | hunger+20, mood+5, copper-10 |

**旧架构**：目标 → 唯一行动（买炊饼）
**新架构**：目标 → 匹配演出（8种觅食演出中按情境选一个）→ 执行效果

### 演出(Scene)定义

```typescript
interface Scene {
  id: string;                    // 唯一标识
  name: string;                  // 演出名称
  goalCategory: string;          // 匹配的目标类别(survival/social/work/...)
  
  // ═══ 发起者(Actor)条件 — 谁能触发这个演出 ═══
  actorTraits: string[];         // 发起者需要的性格(满足其一,空=不限)
  actorForbiddenTraits: string[];// 发起者不能有的性格
  actorProfession?: string[];    // 发起者职业要求
  actorMinCopper?: number;       // 发起者最低铜钱
  actorMaxCopper?: number;       // 发起者最高铜钱(穷人专属)
  actorEmotion?: string;         // 发起者情绪要求
  actorMinHealth?: number;       // 发起者最低健康
  
  // ═══ 目标(Target)条件 — 需要谁在场才能触发 ═══
  targetRequired: boolean;       // 是否需要目标NPC(单人演出=false)
  targetTraits?: string[];       // 目标需要的性格(满足其一)
  targetProfession?: string[];   // 目标职业要求
  targetMinCopper?: number;      // 目标最低铜钱(如偷窃需要有钱人)
  targetMaxCopper?: number;      // 目标最高铜钱
  targetRelationType?: string;   // 与目标的关系要求(friend/enemy/stranger/any)
  targetMinHealth?: number;      // 目标最低健康
  
  // ═══ 环境条件 ═══
  location?: string[];           // 地点要求
  timeOfDay?: 'day' | 'night' | 'dawn' | 'dusk';  // 时段要求
  weather?: string[];            // 天气要求
  season?: string[];             // 季节要求
  
  // ═══ 结果判定 — 参考《漫野奇谭》成功/失败分支 ═══
  outcomeType: 'certain' | 'contested' | 'chance';
  // certain:  确定成功(如买炊饼，只要有钱就成功)
  // contested: 对抗判定(发起者 vs 目标，如偷窃 vs 警觉)
  // chance:   概率判定(固定概率，如采药是否找到)
  
  contestedStat?: { actor: string; target: string }; // 对抗属性(如 {actor:'cunning', target:'alertness})
  successChance?: number;        // chance类型的成功概率(0-1)
  
  // ═══ 成功分支 ═══
  successNarrative: string;      // 成功叙事模板
  successEffects: Record<string, number>;          // 成功对发起者的效果
  successTargetEffects?: Record<string, number>;   // 成功对目标的效果
  successRelationChange?: number;                  // 成功的关系变化
  
  // ═══ 失败分支 ═══
  failureNarrative?: string;     // 失败叙事模板(certain类型不需要)
  failureEffects?: Record<string, number>;         // 失败对发起者的效果
  failureTargetEffects?: Record<string, number>;   // 失败对目标的效果
  failureRelationChange?: number;                  // 失败的关系变化
  
  // ═══ 权重与冷却 ═══
  weight: number;                // 基础权重(1-10)
  cooldownTicks: number;         // 冷却tick数
}
```

**结果判定规则**：

```
certain（确定型）：
  例：买炊饼、散步、读书
  只要前置条件满足，必定成功
  只需要 successNarrative + successEffects

contested（对抗型）：
  例：偷窃(狡猾 vs 警觉)、打架(勇敢 vs 强壮)、说服(口才 vs 固执)
  发起者和目标各有一个对抗属性
  成功率 = actorStat / (actorStat + targetStat) × 50%~80%
  需要成功+失败两套叙事和效果

chance（概率型）：
  例：采药是否找到好药(30%)、钓鱼是否钓到大鱼(20%)
  固定概率判定
  需要成功+失败两套叙事和效果
```

**关键设计原则（参考《漫野奇谭》）**：

1. **严格前置条件**：每个演出的 `targetRequired` 和目标条件决定了是否可以触发。如果没有符合条件的目标NPC在场，这个演出会被跳过，不会出现在候选列表中。
   - 小偷想偷窃 → 检查附近有没有 `targetMinCopper>20` 的人 → 没有 → 偷窃演出不可用 → 转而执行其他生存类演出
2. **角色对称性**：互动演出对双方都有影响。偷窃成功→小偷得钱/受害者失钱；偷窃失败→小偷被抓/受害者安全。
3. **匹配宽容度**：条件不要太严苛（否则很多演出永远不会触发），也不要太宽松（否则角色行为前后不一致）。参考漫野奇谭的平衡：性格特征满足其一即可，但关键条件（如关系类型、职业）必须精确匹配。

### 演出匹配流程

```
1. NPC 生成目标(goalCategory)
   例：hunger<40 → 目标="觅食"(survival)

2. 从演出库中筛选所有 goalCategory 匹配的演出

3. 逐一严格检查条件（任一不满足则跳过该演出）：
   ├─ actorTraits — 发起者性格是否匹配(满足其一,空=不限)
   ├─ actorForbiddenTraits — 发起者性格是否冲突(有则排除)
   ├─ actorProfession — 发起者职业是否匹配
   ├─ actorMinCopper / actorMaxCopper — 发起者铜钱范围
   ├─ actorEmotion — 发起者情绪是否匹配
   ├─ actorMinHealth — 发起者健康是否足够
   │
   ├─ 【关键】targetRequired=true 时，必须检查：
   │  ├─ 附近是否有满足 targetTraits 的NPC
   │  ├─ 目标NPC是否满足 targetProfession
   │  ├─ 目标NPC是否满足 targetMinCopper / targetMaxCopper
   │  ├─ 目标NPC是否满足 targetRelationType (friend/enemy/stranger)
   │  ├─ 目标NPC是否满足 targetMinHealth
   │  └─ 如果没有符合条件的目标NPC → 此演出不可用，跳过
   │
   ├─ location — 当前地点是否满足
   ├─ timeOfDay — 时段是否满足
   ├─ weather — 天气是否满足
   ├─ season — 季节是否满足
   └─ cooldown — 是否在冷却中(跳过)

4. 如果没有任何演出满足条件 → NPC回退到基础行为(回家/原地等待)

5. 对所有满足条件的演出计算综合权重：
   score = baseWeight
         × (1 + traitMatchBonus)      // 性格匹配+50%
         × (1 + emotionMatchBonus)    // 情绪匹配+30%
         × (1 + relationBonus)        // 关系匹配+40%
         × (1 + aspirationBonus)      // 抱负匹配+20%
         × (0.9 + random * 0.2)       // ±10%随机

6. 按权重随机选择一个演出(softmax温度选择)

7. 结果判定：
   ├─ certain → 直接执行成功分支
   ├─ contested → 对抗判定(actorStat vs targetStat)
   │  └─ 成功率 = actorStat / (actorStat + targetStat + 1) × 调整系数
   └─ chance → 概率判定(与successChance比较)

8. 参数化填充叙事模板并执行对应分支的效果
```

**目标选择优先级**（当多个NPC满足目标条件时）：
1. 关系最近的NPC优先（朋友>熟人>陌生人）
2. 距离最近的NPC优先
3. 随机选择

### 参数化叙事模板

叙事模板支持以下插值变量，运行时由系统自动填充：

| 变量 | 来源 | 示例值 |
|------|------|--------|
| `{npcName}` | NPC名称 | "王掌柜" |
| `{targetName}` | 目标NPC名称 | "张三" |
| `{location}` | 当前地点 | "东市" |
| `{weather}` | 天气描述 | "阳光正好" |
| `{timeDesc}` | 时段描述 | "正午时分" |
| `{profession}` | 职业 | "商贩" |
| `{item}` | 相关物品 | "苏绣" |
| `{emotion}` | 当前情绪 | "心满意足" |

**叙事设计原则**：
1. **留白**：只描述可观察行为，不解释内心动机（让玩家脑补）
2. **感官细节**：用视觉/听觉/嗅觉让叙事更有画面感（"热腾腾""吱呀""炊饼香"）
3. **性格驱动**：同一个目标，不同性格的NPC叙事完全不同（勤劳→赶早市 vs 懒散→睡懒觉）
4. **关系影响**：有目标NPC参与时，叙事自然融入关系（"挤了挤眼""交换眼神"）

### 冷却与去重

- 每个演出有 `cooldownTicks`（默认5tick），同一NPC在冷却期内不会再次触发同一演出
- 系统记录每个NPC最近执行过的演出ID及tick
- 相邻两次演出的ID不能相同（强制多样性）

### 与现有系统的集成

演出库系统与以下系统协同工作：

```
NPC Tick 流程：
  1. 个性化需求衰减（性格影响衰减率）
  2. 更新压力系统（需求→压力）
  3. 更新情绪（需求+关系触发）
  4. 更新关系（同区域+交互）
  5. 心愿刷新（情绪+性格）
  6. 每日计划（性格hash）
  7. 抱负系统（职业倾向）
  8. 隐藏特征（影响决策偏好）
  9. 生成目标（基于最紧迫需求）
  10. 【演出匹配】← 从演出库匹配最佳演出
  11. 执行演出效果 + 参数化叙事
  12. 链式反应检查（演出结果触发连锁事件）
  13. 生成叙事片段（可观察行为描述）
  14. 记忆固化（短期→长期记忆→人格改变）
```

**第10步是新增的演出匹配环节**，替代原来"目标→固定行动"的直接映射。

### 涌现效果示例

#### 同一个"觅食"目标，不同NPC的差异化表现

**王掌柜（勤劳+精明，铜钱80）**：
→ 匹配演出 `s_feast`（酒楼大餐）→ "推开酒楼的门，叫了一桌好菜。"

**张三（狡猾+暴躁，铜钱5）**：
→ 匹配演出 `s_lucky_find`（捡到残食）→ "在墙角发现了半块被人丢弃的炊饼。犹豫了一下，还是拿起来吃了。"

**周农夫（勤劳+善良，在城外）**：
→ 匹配演出 `s_wild_forage`（采摘野果）→ "在路边发现了野果，虽然酸涩，但聊胜于无。"

**刘寡妇（好客+健谈，有朋友在场）**：
→ 匹配演出 `s_friend_meal`（找朋友蹭饭）→ "笑嘻嘻地凑到李大夫身边：'吃了吗？没吃一起呗！'"

#### 链式涌现示例

```
1. 张三(混混)执行演出"暗巷偷窃" → 成功偷了李大夫的钱袋
2. 链式反应触发：
   - 李大夫获得记忆"被偷"(强度8, 情绪:愤怒)
   - 李大夫与张三关系-30(变为仇敌)
   - 李大夫告诉在场的人(传播事件)
   - 周围人与张三关系-5
3. 下一tick：
   - 李大夫匹配演出"交头接耳"(因愤怒情绪+狡猾性格)
     → 和王掌柜说了被偷的事
   - 钱捕头(衙役)执行"认真巡逻"
     → 注意到张三形迹可疑，触发追捕
   - 张三执行"躲入暗处"(因被追+胆小)
     → "闪身躲进一条暗巷，贴着墙屏住呼吸"
4. 长期效果：
   - 李大夫的记忆"被偷"固化(强度8 > 最弱长期记忆)
   - 李大夫性格改变：获得"多疑"特征
   - 此后李大夫对陌生人的演出匹配倾向于"交头接耳"而非"茶楼闲谈"
```
