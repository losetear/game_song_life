# 08 — 技术架构（万级对象版）

## 设计目标

| 指标 | 目标值 |
|------|--------|
| 同时存在的实体总数 | ≥ 10,000 |
| 其中NPC | 500-1000 |
| 其中动物 | 500-2000 |
| 其中植物/矿物 | 3000-5000 |
| 其中建筑/物品 | 2000-3000 |
| 玩家感知范围内的活跃实体 | ≤ 50 |
| 单次操作的服务器计算时间 | ≤ 150ms |
| 端到端响应延迟 | ≤ 250ms |
| 存档体积 | ≤ 1MB（数据库增量存储） |
| 存档读写时间 | ≤ 500ms |
| 同时在线玩家（多人模式预留） | ≤ 100 |
| WebSocket消息大小 | ≤ 10KB/条 |

## 一、实体-组件-系统（ECS）架构

**为什么不用OOP？**
10000个对象各自独立持有属性 → 内存碎片化、缓存命中率极低、无法批量处理。
ECS将数据按类型连续存储，可以高效批量更新。

### 实体（Entity）

实体只是一个ID，不持有任何数据。

```
实体ID = 唯一整数

示例：
  #1001 = 王掌柜
  #1002 = 张三
  #2001 = 一只母鸡
  #3001 = 一棵柳树
  #4001 = 布庄建筑
  #5001 = 一块铁矿
```

### 组件（Component）

组件是纯数据，按类型分开存储：

```
位置组件 Position        { x, y, areaId }
生命组件 Vital           { hunger, fatigue, health, mood }
身份组件 Identity        { name, profession, age, personality[] }
金钱组件 Wallet          { copper }
背包组件 Inventory       { items[] }
AI组件 AI               { goals[], currentPlan[], planCooldown }
记忆组件 Memory          { events[], impressions{} }
好感组件 Relations       { {targetId: score} }
生长组件 Growth          { stage, growProgress, seasonReq }
产出组件 Produce         { itemType, interval, amount }
耐久组件 Durability      { max, current }
天气组件 Weather         { type, intensity, duration }
建筑组件 Building        { type, ownerId, openHours }
交互组件 Interactable    { actions[], conditions{} }
```

**关键设计**：
- 非所有实体都有所有组件。一只鸡没有AI组件和记忆组件，只有Vital + Growth + Produce
- 一块石头只有Position + Interactable + Durability
- 组件的稀疏性 = 性能优势

### 系统（System）

系统是纯逻辑，遍历拥有特定组件组合的实体：

```
系统列表：

VitalSystem      → 遍历有 Vital 的实体，处理饥饿/疲劳/健康自然变化
AISystem         → 遍历有 AI 的实体，运行GOAP规划（仅活跃实体）
GrowthSystem     → 遍历有 Growth 的实体，推进生长进度
ProduceSystem    → 遍历有 Produce 的实体，定期产出
WeatherSystem    → 管理天气状态，影响其他系统
RelationSystem   → 遍历有 Relations 的实体，处理好感衰减
EconomySystem    → 管理物价、供需、交易
SpatialSystem    → 管理空间索引，查询附近实体
PerceptionSystem → 计算玩家可感知的实体列表
SaveSystem       → 管理存档读写
```

### ECS 数据流

```
┌───────────────────────────────────────────────────┐
│                    实体管理器                       │
│  Entity #1001: [Position, Vital, Identity, AI,     │
│                 Wallet, Memory, Relations]         │
│  Entity #2001: [Position, Vital, Growth, Produce]  │
│  Entity #3001: [Position, Interactable, Durability] │
│  Entity #4001: [Position, Building, Interactable]  │
└──────────────┬────────────────────────────────────┘
               │
               ▼
┌───────────────────────────────────────────────────┐
│                  系统调度器（每tick）                │
│                                                    │
│  1. WeatherSystem  → 更新天气                      │
│  2. GrowthSystem   → 更新植物生长                   │
│  3. VitalSystem    → 更新生命状态                   │
│  4. AISystem       → 更新AI行为（仅活跃层）         │
│  5. ProduceSystem  → 处理产出                       │
│  6. EconomySystem  → 更新物价                       │
│  7. RelationSystem → 更新关系                       │
│  8. PerceptionSystem → 更新玩家感知                 │
└───────────────────────────────────────────────────┘
```

## 二、空间索引与分区

### 区域网格

世界划分为区域（Area），每个区域内用网格（Grid）管理实体：

```
世界层级：
  World
  ├── Area: 汴京城内
  │   ├── Grid: 中心大街 (可容纳 ~200实体)
  │   ├── Grid: 州桥夜市
  │   ├── Grid: 官衙
  │   ├── Grid: 寺庙区
  │   ├── Grid: 西市
  │   ├── Grid: 东市
  │   ├── Grid: 码头区
  │   ├── Grid: 城北民宅
  │   ├── Grid: 城南民宅
  │   └── Grid: 作坊区
  ├── Area: 城外农田
  │   ├── Grid: 东郊田地
  │   ├── Grid: 南郊田地
  │   └── Grid: 水渠
  ├── Area: 山林
  │   ├── Grid: 浅山
  │   ├── Grid: 深山
  │   └── Grid: 山涧溪流
  └── Area: 河流
      ├── Grid: 上游
      └── Grid: 河滩
```

### 实体查询

```
查询接口：
  getEntitiesInGrid(gridId)        → 返回该网格所有实体
  getEntitiesInRange(x, y, radius) → 返回范围内的实体
  getEntitiesByComponent(type)     → 返回拥有某组件的所有实体
  getEntitiesByArea(areaId)        → 返回某区域所有实体
```

### 场景实例化

当玩家进入一个Grid时，该Grid内的所有实体被"激活"。
离开时，非关键实体回到"摘要"状态。

## 三、模拟LOD（Level of Detail）

**核心思想**：不是所有10000个实体都需要同等精度的模拟。

### 三级模拟精度

```
┌─────────────────────────────────────────────────────────────┐
│                     模拟精度分层                              │
├──────────────┬──────────────────┬───────────────────────────┤
│  L0 完整模拟  │  L1 简化模拟      │  L2 统计模拟              │
│  (玩家周围)   │  (同城但不在视野)  │  (城外/偏远)              │
├──────────────┼──────────────────┼───────────────────────────┤
│  GOAP全运行   │  行为树/日程脚本  │  仅统计数值变化            │
│  逐个实体更新  │  批量简化更新     │  区域级聚合更新            │
│  完整记忆系统  │  仅保留关键记忆   │  无记忆                   │
│  实时对话生成  │  预设对话模板     │  无对话                   │
│  完整属性变化  │  简化属性变化     │  仅统计级属性              │
│              │                  │                           │
│  数量: ≤50   │  数量: ≤500      │  数量: ≤10000             │
│  频率: 每动作 │  频率: 每时辰     │  频率: 每天               │
└──────────────┴──────────────────┴───────────────────────────┘
```

### LOD 切换规则

| 触发条件 | 目标LOD |
|---------|---------|
| 实体在玩家当前Grid | L0 |
| 实体在同一Area但不同Grid | L1 |
| 实体在不同Area | L2 |
| 实体距玩家移动方向较远 | 降一级 |
| 特殊事件波及（瘟疫/粮荒） | 相关实体临时升级 |

### 各实体类型的默认LOD

| 实体类型 | 默认LOD | 说明 |
|---------|---------|------|
| 玩家 | L0 (恒定) | 始终完整模拟 |
| 命名NPC | L0/L1 | 玩家附近=L0，否则L1（永远不低于L1） |
| 通用NPC | L0/L1/L2 | 按距离正常降级 |
| 家畜 | L0/L1 | 玩家附近=L0，否则L1 |
| 野生动物 | L0/L2 | 玩家附近=L0，远处=L2（只统计数量） |
| 农作物 | L0/L2 | 玩家田地=L0，其余=L2（区域统计产量） |
| 野生植物 | L0/L2 | 玩家采集范围=L0，远处=L2 |
| 建筑 | L0/L1 | 始终至少L1（需响应经营状态） |
| 矿物 | L0/L2 | 玩家附近=L0，远处=L2 |
| 天气 | L0 (全局) | 天气是全局系统，始终完整模拟 |

## 四、分层AI系统

### 原GOAP设计的问题
10000个实体如果都跑GOAP规划器：
- 每个规划器计算约0.5-5ms
- 10000 × 1ms = 10秒/次更新
- 完全不可行

### 新方案：三层AI

```
┌─────────────────────────────────────┐
│         L0: GOAP（完整规划）          │
│  仅用于：玩家视野内的命名NPC          │
│  数量：≤ 10                          │
│  频率：每次玩家动作                   │
│  能力：完整目标规划、记忆驱动决策      │
├─────────────────────────────────────┤
│         L1: 行为树（简化决策）         │
│  用于：玩家同城的NPC + 家畜           │
│  数量：≤ 500                         │
│  频率：每时辰                         │
│  能力：按职业模板+随机分支选择行为      │
├─────────────────────────────────────┤
│         L2: 统计模拟（无个体AI）       │
│  用于：城外的所有实体                 │
│  数量：≤ 10000                       │
│  频率：每天                           │
│  能力：按区域统计产量/消耗/人口变化     │
└─────────────────────────────────────┘
```

### L1 行为树模板

```
行为树节点类型：
  Sequence（顺序执行）
  Selector（选择执行）
  Random（随机分支）
  Condition（条件检查）

[商贩行为树]
  Selector
  ├── Condition: 饥饿 > 70 → Sequence: 去买饭 → 吃饭
  ├── Condition: 疲劳 > 80 → Sequence: 回家 → 休息
  ├── Condition: 时辰 ∈ [辰,酉]
  │   └── Selector
  │       ├── Condition: 库存充足 → 在铺子卖货
  │       ├── Condition: 库存不足 → 去进货 → 回铺子卖货
  │       └── 随机(20%) → 去茶馆社交
  ├── Condition: 时辰 ∈ [戌,亥] → 去夜市/回家
  └── 默认：在家休息

[家畜行为树]
  Selector
  ├── Condition: 饥饿 > 60 → 觅食
  ├── Condition: 天黑 → 回窝
  ├── Random(30%) → 游荡
  └── 默认：待在原地
```

### L2 统计模拟

```
区域级模拟（每天更新一次）：

[城外农田区域]
  输入：面积、农夫数量、天气、季节
  计算：
    产量 = 基础产量 × 季节修正 × 天气修正 × 劳力修正
    消耗 = 农夫数量 × 每日消耗
    净产出 = 产量 - 消耗
  输出：
    市场粮食供给量 += 净产出
    区域人口 = 按粮食余量波动

[山林区域]
  输入：动物数量上限、季节、狩猎压力
  计算：
    繁殖量 = 当前数量 × 繁殖率 × 季节修正
    狩猎量 = 猎人数量 × 狩猎效率
    当前数量 = max(0, 上日数量 + 繁殖量 - 狩猎量)
  输出：
    可狩猎动物数量
    可采集植物数量
```

## 五、玩家感知系统

**问题**：10000个实体不可能全部展示给玩家。
**解决**：玩家有感知半径，只接收范围内实体的信息。

### 感知规则

```
感知半径：
  视觉：当前Grid + 相邻Grid
  听觉：当前Grid（可听到隔壁的重大事件）
  嗅觉：当前位置（饭菜香、腐臭）
  消息：通过NPC传播，延迟数小时到数天

感知过滤：
  即使在视觉范围内，也不是所有细节都显示
  - 10步内：看到详细外观和动作
  - 同Grid：看到人物和主要动作
  - 相邻Grid：只看到轮廓和重大事件
```

### 信息传播系统

```
事件传播链：

  张三在夜市偷东西 →
  目击者（3-5个L0/L1 NPC）立刻知道 →
  目击者告诉茶馆里的人（1-2时辰延迟）→
  茶馆传遍（半天）→
  全城皆知（1-2天）

  传播速度取决于：
  - 事件重大程度
  - 目击者数量
  - 目击者社交活跃度
  - 是否涉及知名人物
```

## 六、程序化生成

### 为什么必须程序化？

10000个实体不能全部手写。需要模板+规则自动生成。

### 生成策略

```
实体生成流程：

1. 世界初始化
   ├── 区域定义（手写）: 区域类型、面积、容量上限
   ├── 建筑布局（手写+规则）: 固定建筑手写，填充建筑按规则生成
   └── 初始实体生成（程序化）: 按区域容量和类型比例生成

2. 命名NPC（手写 + 程序辅助）
   ├── 核心NPC：10-20个，完全手写（名字、性格、故事）
   └── 半手写NPC：50-100个，名字+职业手写，细节程序生成

3. 通用NPC（完全程序化）
   ├── 从模板库抽取：名字库+外貌库+性格库+职业库
   ├── 组合生成：随机组合确保不重复
   └── 数量：按区域经济容量动态调整

4. 动物（程序化 + 规则）
   ├── 家畜：按农户/养殖场数量分配
   └── 野生动物：按山林区域生态容量生成，有繁殖/死亡循环

5. 植物（程序化 + 生态规则）
   ├── 农作物：按季节和农田面积生成
   └── 野生植物：按区域类型和季节生成，可采集后重生

6. 矿物（程序化 + 固定节点）
   ├── 矿点：固定位置，采完后按规则再生
   └── 石料：按区域类型分布

7. 天气（系统级，非实体）
   └── 按季节+随机+区域生成，全局影响
```

### 模板库示例

```
名字库：{姓: [王,李,张,赵,...], 名: [大,二,三,...], 绰号: [铁牛,瘦猴,...]}
外貌库：{体型: [壮,瘦,中等], 特征: [疤,痣,麻子,...], 穿着: [布衣,绸衫,...]}
性格库：{标签: [勤劳,懒惰,贪婪,善良,...], 权重: 2-3个标签组合}
职业库：{类型: [农夫,商贩,工匠,...], 日常行为树模板, 收入范围}
对话库：{场景: [打招呼,交易,闲聊,...], 模板: "今天{天气_adj}，{问候语}"}
```

## 七、存档策略

### 增量存档

```
存档数据分层：

1. 静态数据（不存，从代码重建）
   ├── 区域定义、建筑布局
   ├── 行动表、行为树模板
   └── 模板库（名字、对话等）

2. 半静态数据（每次变化时存）
   ├── 世界时间、当前天气、季节
   ├── 经济状态（物价、供需）
   └── 区域统计（人口、产量）

3. 动态数据（每次自动存档时存）
   ├── 玩家完整状态
   ├── L0/L1实体完整状态（≤550个）
   └── 全局事件队列

4. 冷数据（仅在手动存档时完整保存）
   ├── L2实体状态（仅存与默认值的差异）
   └── 历史记录
```

### 存档大小估算

```
玩家状态：      ~2KB
L0 NPC (10个)： ~5KB × 10 = 50KB
L1 NPC (500个)：~1KB × 500 = 500KB（简化状态）
L2 差异：       ~200KB（仅存偏移量）
全局状态：       ~50KB
总计：          ~800KB（远低于5MB目标）
```

```
│                         │                                │
└─────────────────────────┼────────────────────────────────┘

                  ┌───────────────┐
                  │   数据库       │
                  │  持久化存储    │
                  └───────────────┘
```

### 为什么改用CS架构

| 理由 | 说明 |
|------|------|
| 多人模式扩展 | 服务器天然支持多玩家共享同一世界 |
| 作弊防护 | 服务器权威计算，客户端无法篡改游戏逻辑 |
| 计算能力 | 服务器可用多核+多线程处理万级实体 |
| 持久化 | 数据库比localStorage可靠，支持大规模数据 |
| 热更新 | 修复bug/调整平衡无需玩家更新客户端 |
| 单机模式 | 初期仍支持单人游玩，只是计算放在服务器 |

### 技术栈选择

```
服务端：
  ├── 语言：TypeScript (Node.js)
  │   └── 与前端同语言，减少上下文切换
  │   └── 生态成熟，WebSocket/数据库库丰富
  │
  ├── 框架：
  │   ├── Fastify / Express — HTTP API
  │   ├── ws / Socket.IO — WebSocket 实时通信
  │   └── 自建 ECS 框架 — 世界模拟
  │
  ├── 数据库：
  │   ├── SQLite（初期/单机）— 零配置，文件级存储
  │   └── PostgreSQL（多人/生产）— 关系型，JSON支持好
  │
  └── 部署：
      ├── Docker — 容器化部署
      └── 初期可单机运行（server和client同机器）

客户端：
  ├── 语言：TypeScript
  ├── 框架：Vue 3 / React（看偏好）
  ├── 通信：WebSocket（实时） + HTTP（非实时操作）
  ├── 离线缓存：IndexedDB（存最近场景数据）
  └── UI：纯DOM渲染（文字游戏无需Canvas）
```

## 九、通信协议

### WebSocket 消息格式

```
所有消息使用 JSON 格式：

// 客户端 → 服务器
{
  "type": "action",
  "actionId": "buy_wotou",
  "params": { "shopId": 1001, "quantity": 2 },
  "seqId": 42  // 客户端序列号，用于匹配响应
}

// 服务器 → 客户端（响应）
{
  "type": "actionResult",
  "seqId": 42,
  "success": true,
  "data": {
    "scene": "你花了20文买了两个炊饼。王掌柜笑着说：'好嘞，拿好嘞！'",
    "stateChanges": {
      "copper": -20,
      "inventory": { "炊饼": 2 }
    },
    "options": [
      { "id": "chat_wang", "text": "和王掌柜闲聊", "cost": "0.3时辰" },
      { "id": "buy_more", "text": "继续买东西" },
      { "id": "leave", "text": "离开布庄" }
    ]
  }
}

// 服务器 → 客户端（世界事件推送）
{
  "type": "worldEvent",
  "event": {
    "text": "远处传来一阵喧哗，似乎有人在喊'抓贼'！",
    "importance": 2,
    "source": "nearby"
  }
}
```

### 通信流程

```
┌────────┐                              ┌────────┐
│ 客户端  │                              │ 服务器  │
└───┬────┘                              └───┬────┘
    │                                       │
    │  1. 建立WebSocket连接                  │
    │──────────────────────────────────────▶│
    │                                       │
    │  2. 同步：服务器推送玩家当前场景完整数据  │
    │◀──────────────────────────────────────│
    │                                       │
    │  3. 玩家点击按钮"买炊饼"               │
    │──────────────────────────────────────▶│
    │     {type: "action", actionId: ...}   │
    │                                       │
    │                          4. 服务器执行  │
    │                          ECS计算结果   │
    │                          AI更新        │
    │                          事件检测      │
    │                                       │
    │  5. 返回结果 + 新选项                   │
    │◀──────────────────────────────────────│
    │                                       │
    │  6. （可选）推送世界事件                 │
    │◀──────────────────────────────────────│
    │     {type: "worldEvent", ...}         │
    │                                       │
    │  7. 玩家点击新选项                      │
    │──────────────────────────────────────▶│
    │     循环 3-6...                        │
```

### 数据传输优化

```
减少带宽占用的策略：

1. 增量更新
   不发送完整场景，只发送变化部分
   客户端缓存最近场景，收到增量后合并

2. 选项预取
   服务器在返回结果时，预计算可能的下一组选项
   减少往返次数

3. 批量推送
   多个世界事件合并为一条消息
   每100ms最多推送一次

4. 数据压缩
   WebSocket消息启用permessage-deflate压缩
   文本压缩率通常可达60-80%
```

### Tick 循环（操作驱动，非自动运转）

```
服务器不自动运转世界。只在玩家操作时触发一次完整更新。

每次玩家操作触发的更新流程：
  详见 10-世界运转与信息传播.md

  1. 接收玩家操作 (1ms)
  2. 计算时间推进量
  3. 执行玩家操作 → 更新ECS (5ms)
  4. L0/L1/L2 各层全面更新 (127ms)
  5. L2叙事引擎生成远方事件
  6. 信息传播引擎计算消息推送
  7. 计算玩家感知范围 (2ms)
  8. 组装响应（场景+附近事件+远方消息）(10ms)
  9. 推送结果到客户端 (1ms)
  10. 自动存档检查 (异步，不阻塞)

  总计: ~150ms/操作

  玩家不操作 → 服务器空闲，世界冻结，不消耗CPU
```

### 后台更新（操作驱动版）

世界不自动运转，但玩家每次操作会按消耗时长推进对应量的世界变化：

```
时间推进量与更新范围：

短操作（< 1时辰，如买东西、聊天）：
  → L0 区域完整更新
  → L1 区域快速更新（仅检查是否有重大事件）
  → L2 不更新
  → 信息传播推进一小步

中等操作（1-3时辰，如干活、等半天）：
  → L0 区域完整更新
  → L1 区域完整更新
  → L2 区域快速更新
  → 信息传播正常推进

长操作（半天以上，如等待一天）：
  → 全部L0/L1/L2完整更新
  → 触发跨日结算（如果跨日）
  → 信息传播完整推进
  → 生成当日事件摘要

跨季节操作（如连续等待多日）：
  → 每日完整更新
  → 触发跨季节结算
  → 生成季节性大事件
  → 远方消息大量涌入
```

## 十一、数据库设计

### 表结构

```
核心表：

worlds（世界实例）
  ├── id, name, created_at, tick_count, current_time
  ├── season, weather, global_events
  └── economy_snapshot (JSON)

players（玩家角色）
  ├── id, world_id, name, age, origin
  ├── attributes (JSON): {体质, 才学, 声望, 运势, 人脉}
  ├── vitals (JSON): {饥饿, 疲劳, 健康, 心情}
  ├── skills (JSON): {种植: 35, 烹饪: 20, ...}
  ├── copper, inventory (JSON)
  ├── position_x, position_y, area_id, grid_id
  └── status_effects (JSON)

entities（所有实体 - ECS核心表）
  ├── id, world_id, entity_type (npc/animal/plant/mineral/building)
  ├── components (JSONB): 按需存储各组件数据
  ├── area_id, grid_id
  ├── lod_level (0/1/2)
  ├── is_active (boolean)
  └── updated_at

npcs（NPC扩展数据 - 仅命名NPC和有AI的NPC）
  ├── entity_id (FK → entities)
  ├── name, profession, personality (JSON)
  ├── goals (JSON), plan (JSON)
  ├── memory (JSONB)
  └── relations (JSONB): {targetId: score}

saves（存档槽）
  ├── id, player_id, slot_name
  ├── snapshot (JSONB): 完整世界快照
  ├── created_at
  └── play_time

area_stats（区域统计 - L2数据）
  ├── area_id, world_id
  ├── population, food_supply
  ├── wildlife_counts (JSON)
  ├── resource_density (JSON)
  └── updated_at

events（事件日志）
  ├── id, world_id, timestamp
  ├── event_type, importance
  ├── description
  ├── involved_entities (JSON)
  └── propagation_state (JSON)
```

### 存档策略（CS版）

```
自动存档：
  → 每游戏日结束时，后台异步写入数据库
  → 不阻塞游戏循环
  → 增量写入（只写变化的实体）

手动存档：
  → 玩家点击"存档"
  → 服务器立即触发完整快照
  → 写入saves表

读档：
  → 从saves表读取快照
  → 恢复所有实体状态
  → 重新初始化世界模拟

存档体积估算（数据库中）：
  玩家状态:       ~2KB
  活跃实体(500):  ~500KB
  L2统计:         ~50KB
  事件日志:       ~100KB
  总计:           ~650KB/档

  100个存档槽 = 65MB → 数据库轻松应对
```

## 十二、多人模式预留

### 当前（单人）模式

```
单人模式架构：
  ┌────────────┐    WebSocket     ┌──────────┐
  │   浏览器    │◀───────────────▶│  服务器   │
  │  (玩家1)   │    localhost     │ (单世界)  │

  服务器可以本地运行（同一台电脑）
  也可以远程运行（云服务器/Docker）
```

### 未来多人模式

```
多人模式架构（仅需少量改造）：
  ┌────────────┐                  ┌──────────────┐
  │   浏览器    │◀─────┐           │              │
  │  (玩家1)   │      │           │              │
  └────────────┘      │           │  游戏服务器    │
                       ├─────────▶│              │
  ┌────────────┐      │  WebSocket│  (共享世界)   │
  │   浏览器    │◀─────┘           │              │
  │  (玩家2)   │                  │              │
  └────────────┘                  └──────────────┘

多人模式需要的改造：
  ├── 感知系统扩展：每个玩家独立的感知范围
  ├── 冲突处理：两个玩家同时与同一NPC交互
  ├── 聊天系统：玩家间通信
  ├── 经济隔离：是否共享经济系统
  └── 安全性：防作弊、防恶意操作

CS架构天然支持多人：
  ✓ 服务器是权威 — 客户端无法作弊
  ✓ WebSocket支持多连接 — 天然多客户端
  ✓ 共享ECS世界 — 同一世界实例
  ✓ 感知系统按玩家过滤 — 每人只看到自己附近的
```

### 多人模式下的系统变化

| 系统 | 单人模式 | 多人模式变化 |
|------|---------|------------|
| ECS | 单世界实例 | 仍为单世界，多玩家共享 |
| 感知 | 1个玩家视野 | N个玩家视野，分别计算 |
| AI | 围绕1个玩家 | N个玩家的L0合并去重 |
| 经济 | 单人影响 | 多人共同影响供需 |
| 存档 | 单人存档 | 世界存档 + 个人存档 |
| 战斗 | 玩家 vs NPC | 玩家 vs NPC + 玩家 vs 玩家 |

## 十三、性能预算（CS版）

```
服务器端（每次玩家操作）：
  接收+解析请求:     1ms
  ECS操作执行:       5ms
  系统更新:         127ms
    ├── AISystem:   100ms (L0 GOAP×10 + L1 行为树×500)
    ├── VitalSystem:  5ms
    ├── GrowthSystem: 10ms
    ├── EconomySystem: 5ms
    ├── RelationSystem: 5ms
    └── 其他:         2ms
  感知计算:           2ms
  场景生成:          10ms
  序列化+发送:        5ms
  总计:            ~150ms

客户端：
  接收消息:           1ms
  解析+状态更新:      5ms
  DOM渲染:          30ms
  总计:            ~36ms

端到端延迟:
  150ms(服务器) + 网络延迟(~20ms) + 36ms(客户端)
  ≈ 200ms（本地）/ ~250ms（远程）

  对于文字冒险游戏，这个延迟完全可接受
```
