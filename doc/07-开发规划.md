# 07 — 开发规划与里程碑

## 开发阶段

> **重要修正**：GOAP是本游戏的核心卖点，不能放到第三阶段才做。应在MVP阶段就用简化版GOAP驱动NPC，而非后期替换。

### 第零阶段：技术验证（2-4周）

**目标**：验证CS架构 + 万级对象核心可行性，这是整个项目的生死关卡

| 验证项 | 内容 | 通过标准 | 失败则 |
|--------|------|---------|--------|
| CS通信 | Node.js + WebSocket 双向通信 | 消息延迟 < 30ms | 换HTTP长轮询 |
| ECS框架 | 自建轻量ECS，创建10000实体 | 创建 < 100ms，内存 < 50MB | 调研第三方ECS库 |
| 空间索引 | 20个Grid，快速查询附近实体 | 查询 < 1ms | 换用简单区域遍历 |
| L0 GOAP | 10个NPC + 30个行动的GOAP | 单次规划 < 10ms | 改为L0也用行为树 |
| L1 行为树 | 500个NPC批量更新 | 批量更新 < 20ms | 减少L1 NPC数量 |
| L2 统计 | 10个区域统计模拟 | 更新 < 5ms | 简化统计模型 |
| 数据库 | SQLite增删改查10000实体 | 写入 < 500ms | 换PostgreSQL或内存优先+异步落盘 |
| 端到端 | 浏览器点击到看到结果 | ≤ 250ms | 优化序列化或减少系统更新 |
| UI原型 | 一个场景 + 文字 + 按钮 | 渲染 < 50ms | 优化渲染策略 |

> **如果第零阶段多项验证失败，需要重新评估项目可行性或降低实体规模目标。**
>
> **架构详见 [11-技术架构.md](11-技术架构.md)**

### 第一阶段：核心骨架（MVP）

**目标**：能跑起来的最小可玩版本

**MVP可玩标准**：能在城里走动、买东西、干活赚钱、吃饭维持生存、度过几天。

## 核心技术挑战

### 1. GOAP实现
- 每个NPC独立运行规划器，性能是关键
- 需设计高效的世界状态表示
- 行动库需要合理抽象，避免组合爆炸

### 2. 世界模拟一致性
- NPC之间的交互需要因果一致
- 经济系统（供需关系）需要平衡
- 时间推进时所有NPC同步更新

### 3. 内容量
- 大量文本描述需要编写
- 每个场景、每个NPC都需要丰富的交互文本
- 需要建立模板+变量系统减少重复编写

### 4. 存档系统
- 需要保存完整世界状态（所有NPC、物品、关系）
- 自动存档不能卡顿
- 永久死亡意味着存档设计要谨慎（防止作弊也要防误操作）

## 有待讨论的开放问题

1. **美术资源**：是否需要场景插图？还是纯文字？
2. **音乐音效**：是否需要？外包还是素材？
3. **多语言**：是否考虑英文版？
4. **多人模式**：未来是否考虑多人在线？
5. **MOD支持**：是否开放让玩家自定义内容？
6. **商业化**：免费+内购？买断制？众筹？

## 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| GOAP性能问题 | 大量NPC同时规划导致卡顿 | 高 | 第零阶段先验证；备选方案：简化行为树 |
| 内容量不足 | 游戏体验重复无聊 | 高 | 建立模板+变量系统；优先保证核心场景深度 |
| 经济系统失衡 | 通货膨胀/紧缩 | 中 | 建立模拟测试环境；设计多层级金钱消耗 |
| 永久死亡劝退 | 玩家投入大量时间后死亡流失 | 高 | 遗产系统（后代继承）；早期死亡有"早夭"特殊反馈 |
| 开发周期过长 | 独立开发资源有限 | 高 | 严格按阶段交付，MVP先行；砍功能优于延期 |
| 核心循环不有趣 | 玩家不知道每天该干什么 | 中 | 在MVP阶段就验证核心循环是否有趣 |
| GOAP涌现出bug | NPC做出荒谬行为 | 高 | 限制行动库；增加行为合法性校验；大量测试 |

### 降级方案

根据第零阶段验证结果，有以下降级路线：

**如果GOAP性能不达标**：
1. L0也改用行为树（最复杂的10个NPC用深度行为树）
2. 所有NPC统一用行为树，保留随机分支增添变化性

**如果10000实体太多**：
1. 降至5000实体（NPC 300 + 动物 700 + 植物/矿物 2000 + 建筑/物品 2000）
2. 降至3000实体（减少远郊区域的实体密度）
3. 动态调整：根据设备性能自动降低L2实体数量

**如果纯前端性能不够**：
1. 使用Web Worker分离模拟线程和UI渲染
2. 关键计算用WASM加速
3. 考虑引入轻量后端分担模拟
